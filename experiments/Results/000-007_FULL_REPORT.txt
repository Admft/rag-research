3. SCORING METHODOLOGY
========================================================================

Source code: src/scoring.py, src/pipeline.py, src/prompts.py

All full-pipeline runs use the same scoring pipeline unless noted.
Runs 006–007 use qwen2.5:14b for both generation and judging.
Run 001 is retrieval-only and has no final_score.

------------------------------------------------------------------------
3.1 Retrieval metrics (Recall@k, MRR@k)
------------------------------------------------------------------------

Computed without any LLM. For each eval question:

  expected_source  — PDF filename from data/eval/questions.jsonl
  retrieved        — top-k chunks returned by the retriever (after reranker)
  found_rank       — 1-based rank of expected_source in retrieved list, or null
  recall_hit       — true if found_rank <= k

Aggregate across N questions:

  Recall@k = (1/N) × Σ [1 if recall_hit else 0]
  MRR@k    = (1/N) × Σ [1/found_rank if found_rank else 0]

Reported in REPORT.txt as percentages (e.g. 98.3%). These metrics answer:
"Did we fetch the right paper?" They do NOT measure answer quality.

Important: Recall@k and MRR@k are reported alongside final_score but are
NOT weighted into it. A run can have excellent retrieval and poor generation.

------------------------------------------------------------------------
3.2 Full-pipeline flow (per question)
------------------------------------------------------------------------

  1. Retrieve top-k chunks
  2. Generate answer (llama3.1:8b or qwen2.5:14b via Ollama)
  3. Parse answer text for scoring
  4. LLM judge → four sub-scores (0–100 each)
  5. Citation parser → citation_accuracy (0–100)
  6. Weighted sum → final_score

Latency in REPORT.txt is retrieve + generate + judge wall-clock time.

------------------------------------------------------------------------
3.3 Answer parsing (before scoring)
------------------------------------------------------------------------

Runs 006–007 (strict_context_json):
  • Ollama JSON schema enforces scratchpad + answer + citations keys
  • parse_generation_response() extracts answer; normalize_citations()
    rewrites [Document N], [N] → [Doc N]
  • answer_parse_rate = 100% in both runs (no token-budget cutoffs)

Runs 003–005 (strict_context_with_citations):
  • Only text inside <answer>...</answer> is scored
  • extract_final_answer() in src/prompts.py strips scratchpad/reasoning
  • answer_parse_rate = fraction of questions with a non-empty parsed answer

Run 000 (og_strict):
  • No XML blocks; full model output is scored as-is
  • answer_parse_rate is 100% if any non-empty answer is returned

Run 002:
  • No <answer> tags yet; extract_final_answer() returns full text unless
    scratchpad-like markers are detected (CoT leak caused scoring noise)

------------------------------------------------------------------------
3.4 LLM judge (four metrics, 0–100 each)
------------------------------------------------------------------------

Model: llama3.1:8b (000, 002–005) or qwen2.5:14b (006–007), Ollama JSON mode.
Runs 006+ use config.judge (separate from generator).

The judge prompt includes:
  • question, expected_answer, expected_source
  • full retrieved context (XML <Document> blocks)
  • parsed generated answer

Required JSON output:

  {
    "answer_correctness": <0-100>,
    "faithfulness": <0-100>,
    "context_recall": <0-100>,
    "context_precision": <0-100>
  }

Definitions sent to the judge:
  answer_correctness — match to hand-written expected answer
  faithfulness       — answer supported by retrieved context only
  context_recall     — retrieved chunks contain info for expected answer
  context_precision  — retrieved chunks relevant, low noise

Failure handling:
  1. First judge call
  2. Retry with "output JSON only" reminder
  3. Fallback heuristics: word overlap between expected/generated answer
     and between expected answer and context (flagged judge_fallback in data.json)

Judge scores are clamped to [0, 100].

------------------------------------------------------------------------
3.5 Citation accuracy (deterministic, 0–100)
------------------------------------------------------------------------

Not LLM-judged. Regex scan for [Doc X] in the parsed answer (X = integer).

Scoring rubric (src/scoring.py → estimate_citation_accuracy):

  No [Doc X] found           → 0
  Valid doc ID fraction      → up to 40 points
    (cited ID maps to position 1..k in retrieved list)
  Any cite hits expected_source → +50 points
  All cites only expected_source → +10 bonus
  Maximum                    → 100

Why run 000 scores 0%: og_strict prompt does not require [Doc X] format,
so citation_accuracy is structurally zero regardless of answer quality.
This removes 10% of the final_score ceiling for OG vs grid runs.

------------------------------------------------------------------------
3.6 Final score (weighted average)
------------------------------------------------------------------------

  final_score = 0.35 × answer_correctness
              + 0.25 × faithfulness
              + 0.20 × context_recall
              + 0.10 × context_precision
              + 0.10 × citation_accuracy

Rounded to 2 decimal places per question; REPORT.txt summary is the mean
over all 60 questions.

Weights (SCORE_WEIGHTS in src/scoring.py):
  answer_correctness   35%
  faithfulness         25%
  context_recall       20%
  context_precision    10%
  citation_accuracy    10%

------------------------------------------------------------------------
3.7 How to read a REPORT.txt scores block
------------------------------------------------------------------------

  Final score          — primary end-to-end metric (0–100)
  Answer correctness   — judge: content vs expected answer
  Faithfulness         — judge: grounded in retrieved chunks
  Context recall       — judge: chunks contain needed information
  Context precision    — judge: chunks are on-topic
  Citation accuracy    — parser: [Doc X] format and source alignment
  Answer parse rate    — pipeline: <answer> block (003–005) or JSON parse (006–007)
  Recall@5 / MRR@5    — retrieval only (see 3.1)
  Avg latency          — seconds per question (retrieve+generate+judge)

6. TRANSITION-BY-TRANSITION ANALYSIS
========================================================================

------------------------------------------------------------------------
000 → 001 → 002  (Founding the experiment grid)
------------------------------------------------------------------------

Run 000 is the recreated original lab: naive 120-word chunking, dense
search only, no reranker, simple refusal prompt. It establishes the
"vanilla" reference point described in the OG baseline design—small chunks
for embedding specificity, 30-word overlap as a sentence-boundary safety
net, cosine dense retrieval, top_k=5 (~600 words of context).

Run 001 was a deliberate smoke test: retrieval-only on 2 questions with
chunk 512 and strict_context. It validated the indexing and eval harness
before committing to 60-question full-pipeline runs. It is not scored on
the same final-score rubric.

Run 002 introduced the experiment-grid baseline in one step:
  • Chunk size 512 → 256 (558 index chunks vs OG's 1,273)
  • Added BGE cross-encoder reranker
  • Switched to strict_context_with_citations (CoT + [Doc X] rules)
  • Enabled full pipeline: retrieve → generate → judge

What went UP (000 → 002):
  • Final score:        57.48 → 66.04  (+8.56)
  • Answer correctness: 55.33 → 69.75  (+14.42)
  • MRR@5:              90.6% → 94.7%  (+4.1 pp)
  • Context precision:  65.88 → 71.63  (+5.75)

What went DOWN (000 → 002):
  • Context recall:     77.42 → 69.80  (−7.62)  — judge sees reranked,
    noisier top-5 vs OG's smaller, more numerous chunks
  • Faithfulness:       64.17 → 66.53  (slight +2.36; within noise)
  • Citation accuracy:   0.00 → 38.67  — OG prompt has no cite format;
    38.7% shows CoT planning leaked into scored output
  • Latency:             1.47s → 3.76s  (+155%) — reranker + longer
    prompts + judge calls per question

Critical issue in 002: chain-of-thought "Planning" text leaked into the
scored answer field. The model reasoned in prose ("Here are the facts I
will use…", "**Step 1 - Planning**") instead of producing a clean cited
answer. Citation accuracy suffered because planning and answer were blended.

Parse failure: 1/60 (enhancingrag_q1) — model filled scratchpad with 9
bullets and hit num_predict=2048 before writing </scratchpad> or <answer>.

13. CONCLUSION
========================================================================

Over eight recorded runs (000–007), the pipeline evolved from a minimal
local RAG prototype to a hybrid-retrieval evaluation harness with JSON-
structured generation, mandatory citations, and reproducible reporting.

The llama3.1:8b era (002–005) locked retrieval at 98.3% Recall@5 with
chunk 256 + BGE reranker + dense search, then pushed the final score from
66.04 → ~74 via XML scratchpad isolation and token-budget tuning.

The v2 era (006–007) swapped in hybrid retrieval, qwen2.5:14b, and JSON
schema output. Run 006 regressed to 67.08 when stacked with top_sentences_5
and loose citation formatting. Run 007 recovered and surpassed the llama
ceiling at 83.22 by restoring full context and normalizing [Doc N] cites.

Total gain OG → best: 57.48 → 83.22 (+25.7 points). The two largest steps:
(1) 002→003 output isolation (+4.5), (2) 006→007 context + citation fixes
(+16.1). Hybrid retrieval fixed the last retrieval miss without MRR loss.

Next experiments (per grid.json) should vary one variable at a time against
the run 007 baseline: query_transform (HyDE), embedding (bge-large), or
context_filter ablations (filter_top10). Do not stack changes.
